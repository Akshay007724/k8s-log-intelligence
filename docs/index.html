how it works

The K-Agent Loop

A THINK→ACT→OBSERVE→DECIDE cycle on each request. The brain calls at most 3 MCP tools, compresses responses into context shards, and routes to specialized sub-agents.

01
THINK
Decides which data sources to query. Cold start always fetches pod status + logs. High severity adds metrics.
state-awarelazy eval
02
ACT
Fires MCP tool calls in parallel via asyncio.gather. Enforced at ≤3 total calls per diagnosis. 1.5s per-call timeout.
asyncparallel≤3 calls
03
OBSERVE
Receives compressed ContextShard responses ≤120 tokens each. Never raw logs. Never raw Kubernetes JSON.
≤120 tokensstructured
04
DECIDE
Computes failure gravity score. Routes to rule-based sub-agents for classification and fix generation.
gravity modelrule-based
FAILURE GRAVITY FORMULA
Gravity = recency × 0.35 + restarts × 0.25 + severity × 0.25 + blast_radius × 0.15
Issues surface only when gravity > 0.65 — filters noise, surfaces signal.
architecture

Four Services, One Answer

All services run as independent FastAPI apps inside the cluster, connected via cluster-internal DNS. RBAC is strictly read-only.

ORCHESTRATION LAYER
:8000
context-weaver
K-Agent brain. Runs the THINK→ACT→OBSERVE→DECIDE loop. Exposes POST /diagnose and GET /health with 2s timeout guard.
SUB-AGENTS (sync, in-process)
log_analysis_agent — compresses shards to signal string
failure_classification_agent — regex rule classifier
root_cause_agent — builds IncidentReport
MCP SERVER LAYER
:8001
mcp-k8s-server
Pod status, restart counts, warning events, deployment generation diff via kubernetes-asyncio.
:8002
mcp-logs-server
Fetches last 100 lines, compresses via regex pattern clustering. Returns error counts and entity extraction.
:8003
mcp-metrics-server
Queries metrics.k8s.io for CPU and memory ratios. Flags OOM-imminent (>95%) at severity 0.95.
failure types

What It Detects

Six classified failure modes. Each returns the exact kubectl commands to run.

💀
OOMKilled
CRITICAL
Container exceeded its memory limit. Kernel OOMKiller terminates it, causing exit code 137.
Fix: kubectl set resources deployment/{svc} --limits=memory=512Mi
🔄
CrashLoopBackOff
CRITICAL
Pod crashing repeatedly on startup. Usually bad config, missing secret, or unready dependency.
Fix: kubectl logs {svc} --previous then verify ConfigMap/Secret.
🖼️
ImagePullBackOff
HIGH
Container image cannot be pulled. Tag missing, registry unreachable, or imagePullSecret invalid.
Fix: kubectl describe pod -l app={svc} to inspect registry auth.
🔌
DependencyTimeout
HIGH
ECONNREFUSED or timeout connecting to downstream service — DB, cache, message queue, or external API.
Fix: nc -zv <host> <port> from within the pod.
📉
DeploymentRegression
HIGH
Recent rollout broke pod availability. Generation mismatch or available replicas below desired count.
Fix: kubectl rollout undo deployment/{svc}
❓
Unknown
LOW
Gravity below threshold or signal doesn't match known patterns. System healthy, or needs manual inspection.
Fix: kubectl describe pod -l app={svc}

deployment

Run It Anywhere

Three deployment options — local Python, Docker Compose, or full Kubernetes cluster.

Step 1 — Clone and configure
git clone https://github.com/Akshay007724/k8s-log-intelligence.git
cd k8s-log-intelligence
cp .env.example .env
Step 2 — Start all services
docker compose up --build
Step 3 — Diagnose a pod
curl -s -X POST http://localhost:8000/diagnose \
  -H "Content-Type: application/json" \
  -d '{"service":"payment-svc","namespace":"production"}' \
  | python3 -m json.tool
Step 1 — Install dependencies
pip install -r requirements.txt
Step 2 — Start MCP servers (3 terminals)
# Terminal 1
IN_CLUSTER=false python3 -m uvicorn src.mcp_k8s_server:app --port 8001

# Terminal 2
IN_CLUSTER=false python3 -m uvicorn src.mcp_logs_server:app --port 8002

# Terminal 3
IN_CLUSTER=false python3 -m uvicorn src.mcp_metrics_server:app --port 8003
Step 3 — Start the brain
IN_CLUSTER=false MCP_K8S_URL=http://localhost:8001 \
MCP_LOGS_URL=http://localhost:8002 \
MCP_METRICS_URL=http://localhost:8003 \
python3 -m uvicorn src.context_weaver:app --port 8000
Step 1 — Namespace and secret
kubectl create namespace monitoring
kubectl create secret generic k8s-log-intelligence-secrets \
  --from-literal=anthropic_api_key=YOUR_KEY -n monitoring
Step 2 — Build and push image
docker build -t YOUR_REGISTRY/k8s-log-intelligence:latest .
docker push YOUR_REGISTRY/k8s-log-intelligence:latest
Step 3 — Deploy to cluster
kubectl apply -f kubernetes/
kubectl rollout status deployment/context-weaver -n monitoring
